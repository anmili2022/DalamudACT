## 二、最近遇到的主要问题与解决方案

## 1. 窗口注册 API 不兼容，插件加载阶段可能直接异常

### 问题

在当前 Dalamud 运行时下，窗口系统的 `AddWindow(...)` 签名与本地编译期预期不一致。

已记录异常：

`MissingMethodException: Method not found: 'Void Dalamud.Interface.Windowing.WindowSystem.AddWindow(Dalamud.Interface.Windowing.Window)'`

### 影响

- 插件可能在构造阶段就因为窗口注册失败而崩溃
- 后续 UI 与主窗口逻辑无法正常初始化

### 解决方案

- 在 `DalamudACT/UI/PluginUI.cs` 中改为使用**反射方式**调用 `WindowSystem.AddWindow(...)`
- 不再绑定旧签名
- 只按“方法名 + 单参数数量”做匹配，提高对运行时差异的兼容性

### 当前状态

已解决，成为当前运行环境下的重要兼容修复点。

## 2. 旧版 SigScanner 注入方式失效

### 问题

旧版直接注入 `ISigScanner` / `SigScanner` 的方式，在当前 Dalamud 运行时下不可用。

### 影响

- 部分签名扫描相关能力无法正常初始化
- 影响 Hook 与运行时对接

### 解决方案

- 改为注入 `IGameInteropProvider`
- 通过反射从 `GameInteropProvider` 内部取出 scanner
- 统一通过 `DalamudApi.ScanText(...)` 做签名扫描

### 相关文件

- `DalamudACT/DalamudApi.cs`

### 当前状态

已解决，旧直注方式已被兼容层绕开。

## 3. 启用插件后，在 `Framework.Update` 周期里持续抛异常并拖崩宿主

### 问题

根据 `E:\git\crash\dalamud.log`，已确认本轮崩溃的直接触发点不是 SigScanner，而是：

```text
System.MissingMethodException:
Method not found:
'UInt16 Dalamud.Plugin.Services.IClientState.get_TerritoryType()'
```

触发路径：

- `DalamudACT.ACT.GetPlaceName()`
- `DalamudACT.ACT.OnFrameworkUpdate(IFramework framework)`

### 影响

- 插件虽然加载成功，但在每次 `Framework.Update` 时都会命中异常
- 异常持续抛出后，会把游戏宿主一起拖崩

### 解决方案

- 在 `DalamudACT/DalamudApi.cs` 中增加反射兼容入口：
  - `GetTerritoryTypeId()`
  - `GetLocalPlayerName()`
- 在 `DalamudACT/Plugin/ACT.cs` 中移除对以下成员的直接调用：
  - `ClientState.TerritoryType`
  - `ClientState.LocalPlayer`
- 为 `OnFrameworkUpdate(...)` 增加保护：
  - 首次异常记录日志
  - 后续不让异常继续冲击游戏主循环

### 当前状态

已解决，这是这轮稳定化处理中的关键修复之一。

## 4. 高风险 Hook 影响稳定性，必须优先保证“插件先能跑”

### 问题

某些 Hook 在当前环境下风险较高，尤其是：

- `ActorControlSelf`
- `Cast`

### 影响

- 一旦 Hook 不兼容，容易直接影响插件或游戏稳定性
- 即便统计链路更完整，也不适合在当前阶段优先启用

### 解决方案

- 当前版本只保留最稳的 `ActionEffectHandler.Receive` Hook
- 暂时禁用 `ActorControlSelf` 和 `Cast`
- 采用“兼容优先模式”，先确保插件能稳定加载和运行

### 当前代价

会带来以下已知缺口：

- DoT 归因不完整
- HoT 归因不完整
- Death 统计链路不完整
- 一些依赖 `ActorControlSelf / Cast` 的遭遇生命周期补强逻辑被关闭

### 当前状态

阶段性解决，属于“先保稳定，再逐步恢复能力”的处理方式。

## 6. 统计范围过宽，可能把队伍外对象误算进来

### 问题

之前存在按名字兜底匹配的逻辑，可能把不在当前队伍中的其他对象误计入统计。

### 影响

- DPS 统计口径不可靠
- 队伍内输出和战斗回看结果可能失真

### 解决方案

  - 本地玩家自己
  - `PartyList` 中的当前队友
- 宠物 / 召唤物通过 `OwnerId` 归属到可追踪对象
- 移除之前按名字兜底匹配的逻辑

### 相关文件

- `DalamudACT/Stats/LocalStatsService.cs`

### 当前状态

已解决，统计范围已经与当前设计目标对齐。

## 7. 禁用旧 Hook 后，死亡统计需要补新的实现链路

### 问题

原有死亡统计链路依赖已禁用的 `ActorControlSelf` Hook，禁用后需要替代方案。

### 影响

- `DPS` 面板中的 `死亡` 列可能无法正确反映实时结果
- 总计行也无法稳定汇总全队死亡数

### 解决方案

在 `LocalStatsService.Update(...)` 中改为轮询 `PartyList`：

- 记录每个队友上一次 `CurrentHP`
- 当满足 `previousHp > 0 && currentHp == 0` 时，计一次死亡
- 仅在 `inCombat == true` 或遭遇已开始时生效，避免待机 / 切图误记
- 优先使用 `member.GameObject?.EntityId`
- 取不到时回退到 `member.ObjectId`

### 结果

- 玩家队友与 NPC 队友都可以进入死亡统计
- `总DPS / 全队` 汇总行中的 `死亡` 列可以显示全队死亡总和

### 当前限制

如果插件是在角色已经死亡之后才启用，不会补记那一次历史死亡。

### 当前状态

已解决，属于兼容当前阶段设计的稳定替代方案。

  - 当前选中项高亮显示
- 当新的实时战斗开始时：
  - 自动切回实时数据
  - 清除历史选中态

### 相关文件

- `DalamudACT/UI/StatsModels.cs`

## 10. 测试数据导入会清空历史，且重复导入容易污染记录

### 问题

早期导入测试数据时，会先清空已有历史，且多次导入容易造成重复堆积。

### 影响

- 不利于验证历史记录功能
- 测试操作会破坏已有历史数据
- 切换历史记录时，容易出现大量重复样本

### 解决方案

- 导入测试数据时不再清空已有历史
- 改为：
  - 保留已有历史
  - 将 3 条测试快照追加到历史中
  - 若同一条测试记录已存在，则按 `区域 + 时长` 覆盖更新

### 当前状态

已解决，现在可以重复导入测试数据而不破坏原有历史。

## 21. 旧失败发版任务不能直接重跑，否则仍会沿用旧工作流快照

### 问题

如果旧发版任务失败后直接在 GitHub 上点 `Re-run jobs`，即便仓库中的工作流文件已修好，GitHub 仍可能使用旧的 workflow 快照和旧的 `GITHUB_SHA / GITHUB_REF`。

### 影响

- 会继续沿用旧的错误打包逻辑
- 修过 workflow 以后，重跑旧任务仍可能同样失败

### 解决方案

- 不再对旧失败 release 任务直接使用 `Re-run jobs`
- 需要重建既有 tag 的 release 时，改用 `workflow_dispatch`
- 在 `Create Release` 工作流中，显式传入准确的既有版本 tag

### 当前状态

已形成稳定操作规范，并已写入交接文档与 runbook。
