## 已修复的问题

### 1. TerritoryType 兼容层

- 旧代码直接依赖 `IClientState.TerritoryType`，在当前运行时上存在兼容性风险。
- 已在 `DalamudACT\DalamudApi.cs` 增加 `GetTerritoryTypeId()` 兼容方法。
- `DalamudACT\ACT2.cs` 与 `DalamudACT\PluginUI.cs` 已改为通过兼容层读取 territory。
- 目标是避免运行时因为 API 差异触发 `MissingMethodException` 或同类加载异常。

### 2. 设置窗口过大且无法调整大小

- 移除了会导致设置窗口强制自适应的窗口行为。
- 将设置窗口默认大小调整为更合理的范围，避免第一次打开就过大。
- 保留手动调整大小的能力。

### 3. 设置窗口 `TypeLoadException`

- 实机运行时发现 `ConfigWindow.Draw()` 抛出：
  - `System.TypeLoadException: Could not load type 'WindowSizeConstraints' from assembly 'Dalamud, Version=15.0.0.2'`
- 根因是代码直接使用了 `Dalamud.Interface.Windowing.WindowSizeConstraints`，而当前运行时不存在该类型。
- 最终修复方式：
  - 移除对 `WindowSizeConstraints` 的直接依赖
  - 改为在 `PluginUI.cs` 中使用 ImGui 当前窗口尺寸做手动夹取
  - 保留“默认大小合理、可拖拽改大小、窗口不会超出屏幕太多”的行为

### 4. 卸载 / 重载后的生命周期缺陷

- 日志显示本次过程中存在：
  - `2026-05-04 03:53:03` 卸载 `DalamudACT`
  - `2026-05-04 04:05:39` 重新加载 `DalamudACT`
- 原代码在 `Dispose()` 时没有完整解绑以下内容：
  - `UiBuilder.Draw`
  - `UiBuilder.OpenConfigUi`
  - `/act` 命令
- 这会导致旧实例在卸载后仍可能被回调，属于真实存在的生命周期问题。
- 已修复为：
  - 在 `Dispose()` 中完整解绑事件和命令
  - 增加 `disposeState` / `IsDisposed` 防护
  - 增加 `CancellationTokenSource`，在卸载时取消后台轮询
  - 在 hook 回调、`Framework.Update` 和 ACT MCP polling 路径中加入 disposed / cancel 判断

### 5. 其他 API 15 适配

- 调整了窗口系统调用方式，避免插件在构造阶段因为接口不兼容直接崩溃。
- 处理了多处旧签名与当前 Dalamud 运行时不一致的问题。
- 对部分旧 hook 路径采用了更保守的处理，避免继续触发旧签名扫描失败。
